# Experimental Rust for GPU Assembly on ROCm: A Summary

## 1. Introduction

This document summarizes the investigation into the feasibility and practicality of using Rust for low-level GPU programming on the ROCm platform, with a specific focus on the potential use of inline assembly (`asm!`) for GPU kernels. The goal is to understand how Rust can interface with ROCm at a level comparable to or supplementing existing C++/HIP development.

## 2. `rust-gpu` Project (`rustc_codegen_spirv`)

*   **Goal:** The primary goal of the `rust-gpu` project, and its core component `rustc_codegen_spirv`, is to compile Rust code into SPIR-V (Standard Portable Intermediate Representation). This allows Rust to be used for writing GPU shaders for both graphics pipelines (e.g., Vulkan, Metal, DirectX 12) and compute kernels.
*   **Mechanism:** It functions as a custom backend for the Rust compiler (`rustc`). Instead of generating machine code for a CPU, it generates SPIR-V.
*   **Output:** The direct output of the compilation process is a SPIR-V module (`.spv` file).
*   **Execution on ROCm:** SPIR-V modules generated by `rust-gpu` can be executed on ROCm-supported GPUs via ROCm's Vulkan runtime (which natively consumes SPIR-V) or its OpenCL runtime (which can also ingest SPIR-V).
*   **Maturity & Status:** The `rust-gpu` project is experimental but under active development. While it has seen good progress, particularly for Vulkan graphics shaders, its use for general-purpose compute kernels is still developing and not yet considered production-ready for all complex scenarios.
*   **Tooling & Debugging:**
    *   **Tooling:** `spirv-builder` is a helper crate to compile Rust GPU code (using `rustc_codegen_spirv`) into SPIR-V modules during a Cargo build process. The `spirv-std` crate provides GPU-specific intrinsics and types, similar to `std` but for the SPIR-V environment.
    *   **Debugging:** Debugging typically relies on the tools provided by the target runtime (Vulkan or OpenCL) and ROCm's own debugging tools (like `rocgdb` or `rocprof`). This often means debugging at the SPIR-V or even the ISA (Instruction Set Architecture) level generated by the driver from SPIR-V, rather than directly at the Rust source level for the GPU code.
*   **Applicability to ROCm Compute (e.g., Transformers):**
    *   For complex compute workloads like transformer models, using `rust-gpu` could be potentially viable in the future for writing specific, performance-critical kernels, especially if a Vulkan or OpenCL host integration model is acceptable.
    *   However, currently, this approach involves a higher effort and risk compared to established HIP C++ development due to the experimental nature, potential for performance unpredictability, and the indirection through SPIR-V. The host application would need to manage the Vulkan/OpenCL runtime, load, and launch these SPIR-V kernels.

## 3. Rust `asm!` Macro for GPU Targets (GCN/CDNA ISA)

*   **Feasibility:** The standard Rust `asm!` macro (or `global_asm!`) is designed for inline assembly on CPU targets (e.g., x86, ARM). It is **not directly supported** by standard Rust tooling for generating code for GPU ISAs like AMD's GCN (Graphics Core Next) or CDNA (Compute DNA).
*   **GPU Code Generation:** GPU code generation for AMD ISAs is a complex process. It typically involves:
    1.  High-level languages (like HIP C++) being compiled by specialized compilers (e.g., HIPCC, which uses Clang/LLVM with an AMDGPU backend).
    2.  Intermediate Representations (like LLVM IR or SPIR-V) being compiled by the GPU driver's JIT (Just-In-Time) compiler into the specific ISA.
*   **Alternative for Raw Assembly:** Writing raw GCN/CDNA assembly is possible but is an "offline" process. Developers would use AMD's assembler tools (part of the ROCm SDK or separate tools like `llvm-mc` with appropriate target settings) to write and assemble `.s` files into GPU machine code objects. These objects would then be loaded and launched from a host language (which could be Rust) using ROCm APIs (e.g., HIP or OpenCL FFI calls to load code objects).
*   **Practicality:** Using Rust's `asm!` macro to directly inline GCN/CDNA assembly for GPU kernels is **currently impractical and not a feature** of the Rust language or its standard `rustc` compiler when targeting AMD GPUs. The compilation pipeline and target architecture are fundamentally different from CPUs where `asm!` is used.

## 4. Mature Alternative: FFI to HIP C++ Kernels

*   **Approach:** This is the most common and mature method for integrating Rust with ROCm for GPU computation.
    1.  Performance-critical kernels are written in HIP C++.
    2.  These kernels are compiled into a shared library or directly linked using the standard ROCm toolchain (HIPCC).
    3.  The Rust application (acting as the host) calls these HIP C++ kernels using Foreign Function Interface (FFI).
*   **Maturity:** This approach is highly mature.
    *   HIP C++ is AMD's primary language for GPGPU programming on ROCm and is well-supported and optimized.
    *   Rust's FFI capabilities are robust and well-documented, allowing seamless calls to C-compatible functions.
*   **Tooling:**
    *   **ROCm SDK:** HIPCC for compiling kernels, `rocgdb` for debugging, `rocprof` for profiling.
    *   **Rust:** Cargo for managing the Rust project, `bindgen` or `cbindgen` tools for automatically generating the FFI bindings between Rust and C/C++.
*   **Example:** Frameworks like `tracel-ai/burn` (a deep learning framework in Rust) utilize this model for their ROCm backend (e.g., via its `cubecl-hip` component), where Rust code orchestrates operations and calls HIP C++ kernels for execution on the GPU.

## 5. Overall Opinion on Practicality for ROCm Development Currently

*   **For writing kernels *directly in Rust*:**
    *   `rust-gpu` (`rustc_codegen_spirv`) is the most promising path. However, it remains experimental and is best suited if a Vulkan or OpenCL host interface is acceptable and the developer is prepared for the current limitations and potential for lower performance or higher development effort compared to HIP C++. It's a space to watch for future developments.
*   **For direct low-level control via *inline assembly in Rust for GPU kernels*:**
    *   This is **not practical or supported** with current Rust tooling for GPU ISAs. Writing raw assembly is an offline process, separate from Rust's `asm!` macro.
*   **For integrating Rust with ROCm for high-performance compute:**
    *   Using Rust as the host language to call HIP C++ kernels (via FFI) is the **most mature, well-supported, and practical approach today.** This leverages the strengths of both ecosystems: Rust for application logic, safety, and modern tooling, and HIP C++ for direct, optimized GPU kernel development with AMD's full backing. This method provides a stable and performant way to build ROCm-accelerated applications using Rust.
