# Architecture of `rust_vector_add`

This document outlines the architecture of the `rust_vector_add` application, a demonstration of using Rust to interact with a C++ HIP kernel for GPU computation.

## 1. Overview

The `rust_vector_add` project aims to show a clear example of Rust controlling GPU operations via AMD's HIP (Heterogeneous-compute Interface for Portability) API. It performs a simple vector addition (`C[i] = A[i] + B[i]`) where the computation itself happens on the GPU, managed by Rust host code.

## 2. Components

The project consists of two main parts: a Rust host application and a C++ HIP GPU kernel.

### 2.1. Rust Host Application (`rust_vector_add` crate)

*   **`src/main.rs`**:
    *   Contains the primary application logic.
    *   Initializes input data (vectors A and B) on the host.
    *   Manages GPU resources:
        *   Allocates device memory for vectors A, B, and C using `hipMalloc`.
        *   Copies input data from host to device (`hipMemcpyHostToDevice`).
        *   Copies results from device to host (`hipMemcpyDeviceToHost`).
        *   Frees device memory using `hipFree`.
    *   Handles the HIP module and kernel:
        *   Loads the pre-compiled GPU kernel (`.co` file) using `hipModuleLoadData`.
        *   Retrieves the specific kernel function (`vectorAdd`) from the module using `hipModuleGetFunction`.
        *   Unloads the module using `hipModuleUnload`.
    *   Launches the GPU kernel using `hipModuleLaunchKernel`, providing necessary parameters (grid/block dimensions, kernel arguments).
    *   Verifies the GPU's results against a CPU computation.
    *   Uses `env_logger` for logging and `scopeguard` for ensuring GPU resources are cleaned up even in case of errors.
*   **`src/hip_utils.rs`**:
    *   Provides safe Rust wrappers and utilities for interacting with the HIP API.
    *   Includes the FFI bindings generated by `bindgen` from `wrapper.h` (via `OUT_DIR/bindings.rs`).
    *   Defines the `HipError` struct for custom error representation.
    *   Features the `hip_call!` macro, which wraps `unsafe` HIP API calls, checks their return status, and converts errors into `anyhow::Result` for idiomatic Rust error handling.
*   **`build.rs`**:
    *   A build script that runs before the main compilation.
    *   Uses the `bindgen` crate to parse `wrapper.h` and generate Rust FFI bindings to the required HIP API functions and types.
    *   Instructs Cargo to link against the HIP runtime library (e.g., `libamdhip64.so`).
*   **`wrapper.h`**:
    *   A C++ header file that includes `hip/hip_runtime_api.h`. This file tells `bindgen` which HIP API elements to generate bindings for.
*   **`Cargo.toml`**:
    *   The Rust package manifest.
    *   Declares dependencies: `anyhow` (error handling), `log` & `env_logger` (logging), `libc` (FFI types), `scopeguard` (resource cleanup), and `bindgen` (build-time FFI generation).

### 2.2. C++ HIP GPU Kernel

*   **`vectorAdd_kernel.hip`**:
    *   A C++ file containing the actual GPU computation logic.
    *   Defines a `__global__` function `vectorAdd(const float* A, const float* B, float* C, int N)`.
    *   This function is executed by many GPU threads in parallel. Each thread calculates `C[i] = A[i] + B[i]` for a specific index `i`.
    *   It's compiled separately using `hipcc` into a Code Object (`.co`) file (e.g., `vectorAdd_kernel.co`).

## 3. Interaction and Data Flow

1.  **Compilation**:
    *   The `vectorAdd_kernel.hip` is compiled by `hipcc` into `vectorAdd_kernel.co`. This is a manual step performed by the user.
    *   The Rust code is compiled by `cargo`. During this process, `build.rs` runs `bindgen` to create the `bindings.rs` file, making HIP APIs callable from Rust.
2.  **Initialization (Rust Host)**:
    *   `main.rs` prepares input vectors `h_A` and `h_B` in host memory.
    *   It allocates corresponding device memory buffers `d_A`, `d_B`, and `d_C` on the GPU.
3.  **Data Transfer (Host to Device)**:
    *   Input data from `h_A` and `h_B` is copied to `d_A` and `d_B` respectively, using `hipMemcpy`.
4.  **Kernel Loading and Execution (Rust Host -> GPU)**:
    *   The `vectorAdd_kernel.co` file's content is read by Rust and loaded onto the GPU using `hipModuleLoadData`.
    *   A handle to the `vectorAdd` function within the loaded module is obtained using `hipModuleGetFunction`.
    *   Rust sets up kernel arguments: pointers to `d_A`, `d_B`, `d_C`, and the vector size `N`.
    *   The kernel is launched on the GPU via `hipModuleLaunchKernel`. The GPU executes the `vectorAdd` function across many threads.
5.  **Data Transfer (Device to Host)**:
    *   After kernel completion (ensured by `hipDeviceSynchronize`), the results from `d_C` are copied back to `h_C_gpu` in host memory using `hipMemcpy`.
6.  **Verification (Rust Host)**:
    *   The GPU results in `h_C_gpu` are compared against a CPU-computed version.
7.  **Cleanup (Rust Host)**:
    *   All allocated GPU resources (device memory, loaded module) are freed. `scopeguard` helps ensure this happens reliably.

## 4. FFI Boundary

The Foreign Function Interface (FFI) is the bridge between Rust and the C-compatible HIP runtime library.
*   **Declaration**: `wrapper.h` declares what C/C++ functions/types are of interest.
*   **Generation**: `bindgen` (via `build.rs`) generates the corresponding `unsafe` Rust function signatures and type definitions.
*   **Invocation**: `hip_utils.rs` (primarily the `hip_call!` macro) invokes these `unsafe` FFI functions.
*   **Safety**: Safe wrappers and error handling (like `hip_call!`) are built on top of the `unsafe` bindings to provide a safer interface to the rest of the Rust application.

## 5. Key Design Choices

*   **`hipModule*` API for Kernel Launch**: Instead of embedding kernel strings or using HIPRTC at runtime, this example uses pre-compiled kernels loaded via `hipModuleLoadData`. This separates kernel compilation from host application runtime and is a common approach for production code.
*   **Error Handling**: Uses `anyhow` for flexible error propagation and `hip_call!` to convert HIP error codes into Rust `Result` types, making error handling consistent.
*   **Resource Management**: Leverages `scopeguard` for RAII-like cleanup of GPU resources, ensuring that `hipFree` and `hipModuleUnload` are called even if errors occur.
*   **Clarity and Simplicity**: The example focuses on demonstrating the core mechanics of Rust-HIP FFI interaction for a basic compute task.

## 6. Build Process Summary

1.  **C++ Kernel Compilation (Manual)**: `hipcc --genco vectorAdd_kernel.hip -o vectorAdd_kernel.co`
2.  **Rust Compilation (Cargo)**:
    *   `build.rs` runs `bindgen` -> generates `OUT_DIR/bindings.rs`.
    *   `cargo` compiles `src/*.rs` files, linking against `libamdhip64.so` and using `bindings.rs`.
    *   Produces the `rust_vector_add` executable.

This architecture provides a foundational model for more complex Rust applications that need to leverage AMD GPU acceleration through HIP.
